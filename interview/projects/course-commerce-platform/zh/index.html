项目深度梳理

课程商业平台

一个面向内部运营团队的 B 端运营管理后台 —— 把课程业务下散落在多个老系统里的商品、营销、用户规则、落地页搭建等所有运营动作，收敛到同一个系统里。 底层是 Umi Max 的集中式路由 + 双层权限体系 + 可复用的 CRUD / 表单抽象 + Swagger 自动生成的 API 契约，把 5 大业务域、20+ 二级页面做到一套交互、一套权限、一套数据口径。

01项目概览

一句话讲清楚这个项目

一个面向内部运营团队的商业化中台，把课程业务下所有和"赚钱"相关的运营操作收敛到同一个后台里： 从商品定义（课程、盒子、优惠券）、营销工具（推广、渠道码、定制活动）、用户规则（限购、画像），到落地页的搭建与发布，形成一条完整的运营闭环。 前端底座用 Umi Max 约定式路由 + access 插件拉齐权限，用 Redux Toolkit 管全局状态、Formily 管复杂表单、ahooks 管异步与交互，API 契约由 Swagger 自动生成，保证前后端类型一致、口径一致。

业务域一览

项目中心
课程项目 / 导航项目 / 模板中心 / 组件库 / 小程序，落地页搭建的入口。
礼品中心
课程商品（供应链 / 商品中心）、实物商品（课程盒子 / 实物盒子）、虚拟商品（优惠券）。
小工具
推广工具、定制活动、渠道活码、排查工具。营销运营的日常家伙事儿。
用户画像
用户限购、课程限购、课程关系，控制哪些用户能买哪些课。
工作台
落地页搭建器（7 步向导 + Formily 表单 + 真机预览）。作为一个入口存在。
背景（S）
课程业务的运营链路散落在多个老后台里 —— 商品在一个系统、优惠券在一个系统、落地页要让前端手写、限购规则靠后端改代码。 运营每做一次活动都要跨 3–4 个系统协同，前端也疲于给各系统打补丁，数据口径和交互都不统一。
目标（T）
把这些散落的运营能力收敛进一个统一的后台，提供一致的交互、一致的权限、一致的数据口径， 并让非技术的运营同学能端到端自助完成大部分日常操作，前端团队从重复交付里抽出身来。
动作（A）
基于 Umi Max 搭建前端底座，约定式路由 + access 插件做菜单级 + 按钮级的双层权限； 业务分层为「业务模块 / 平台能力 / 基础设施」三层，CRUD 场景沉淀成 useTableSearch / useTableSet 等 Hook； 用 Redux Toolkit 管全局状态，Formily 管复杂嵌套表单，ahooks 统一异步与交互； @dandelion/swagger 自动生成 API 契约，axios 拦截器统一接管鉴权、错误和 toast。
结果（R）
一个后台覆盖 5 大业务域、20+ 二级页面；运营跨系统切换的成本大幅降低； 落地页从前端排期变成运营自助，前端团队从「做页面」转向「把平台做好」。

让它真正跑起来的四个关键点

  • 集中式路由 + access 驱动的双层权限。 一份 routes.ts 声明所有路由，每个节点带 access key，Umi Max 自动把「菜单渲染 / 路由拦截 / 按钮级鉴权」三件事一处配置、一处生效。 页面用 <Access>，按钮用 useAuthCheck(authKey)，菜单权限和功能权限明确分层，不会互相污染。
  • CRUD 场景抽象成 Hook，不再复制粘贴。 20+ 列表页只写 columns 和 searchFn，筛选 / 分页 / 排序 / 列持久化 / URL 同步由 useTableSearch + useTableSet 统一接管。 新列表页从 500+ 行压到 ~150 行，交互也天然一致。
  • Formily 扛住复杂嵌套表单。 优惠券、商品这类「深层嵌套 + 字段联动 + 大量校验」的场景，用 Formily schema 声明式描述，响应式联动由 Formily 接管，业务代码只专注数据结构本身。 避免了手写 onChange + 一堆 if 维护表单状态的经典坑。
  • Swagger 自动生成 + axios 拦截器统一收口。 前后端契约由 @dandelion/swagger 基于 OpenAPI 自动生成 TS 类型和请求函数，前端几乎不再手写接口层； axios 拦截器把 401 自动登出 / 跳 SSO、业务码统一 toast、请求头注入全部做掉——业务代码只写 happy path。

几个容易被忽略、但其实很要紧的细节

  • 权限模型明确分层。后端返回的权限列表里 type=0/1 是菜单权限、其它是功能权限；菜单权限驱动 access / 路由、功能权限驱动按钮，两层逻辑在前端入口就拆清楚，后面维护不串线。
  • 多环境构建一套代码四份产物。用 cross-env mode=dev|mock|gray|prod 跑四种构建：mock 让前端脱离后端独立开发、gray 灰度验证、prod 上线。同一份源码、不同 mode 产出不同产物，无需维护多分支。
  • 列表页交互策略统一定义。搜索重置 current、翻页保留筛选、重置清 params、列顺序 / 可见性 localStorage 持久化——这些「默认行为」全部集中在 Hook 内部，20+ 列表页开箱一致，业务开发不用再想一次。

注：本项目的工作台模块里也包含一个低代码落地页搭建器，但团队另有专门的低代码平台作为亮点项目（见 企业级低代码活动搭建平台），这里只做概述、不做深挖。更多业务侧细节见「重点模块」页。
