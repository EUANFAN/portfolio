06Key Decisions

The tradeoffs we made at the key decision points

Early in the project, we made a few choices that shaped everything that came after. This section lays out what we chose, why we chose it, and what we knowingly paid for it.

Decision 1 · Routing: centralized route config, not file-based routing
Problem
Umi Max supports both file-based routing and centralized route config, so we had to choose one.
Choice
Declare all routes centrally in src/routes/index.ts, and let every route carry name / path / component / access.
Why
This was a back-office system, and routes naturally carried permission information. With centralized config, routing, menus, and permissions could be reviewed together in one place. It also made deep menu hierarchies much easier to reason about.
Cost
Adding a menu meant touching both routes and access config. That extra edit cost was real, but it kept menus and permissions aligned.
Decision 2 · Permissions: a two-layer model for menus and functions
Problem
We needed to answer both “can this person see this menu?” and “this person can see the page, but can they click this button?”
Choice
Split permissions into two layers: one for menus and routes, one for buttons and in-page operations. The backend returned one tree, and the frontend split it by type.
Why
The responsibilities stayed clean. Menu permissions controlled whether a page could be entered at all, while function permissions controlled what you could do once you were inside. That made both backend contract design and frontend usage much clearer.
Cost
A permission change might need to be reflected in both places, but that duplication stayed manageable once each person only owned their own business domain.
Decision 3 · State management: Redux Toolkit, not MobX or Context
Problem
We still needed a shared state solution for cross-page data like user info, global config, and troubleshooting context.
Choice
Redux Toolkit, wrapped with typed useAppDispatch and useAppSelector.
Why
The team already understood Redux best, RTK removed a lot of the old boilerplate, TypeScript support was mature, and DevTools debugging stayed straightforward. We did not have a strong enough reason to pay for MobX’s reactive magic or Context’s performance pitfalls.
Cost
It was still heavier than lighter stores like zustand or jotai. In practice we only had three slices, so the weight stayed acceptable.
Decision 4 · Forms: Formily, not a homegrown schema system and not raw AntD Form
Problem
Purchase-limit rules, course relationships, and page-builder config were all deep, nested, highly reactive forms. Raw AntD Form would have meant a lot of ad hoc linkage logic in effects.
Choice
Formily 2 as the standard form solution for all complex forms.
Why
Formily’s reaction model let us pull field linkage back into schema and keep the business code focused on the data structure itself. Compared with building our own schema engine, it saved us from reinventing the wheel; compared with raw AntD Form, the maintainability of linkage and validation was much better.
Cost
The learning curve was real, and new teammates usually needed about a week to feel comfortable with it. The docs were not always perfect either.
Decision 5 · API layer: generated from Swagger, not handwritten
Problem
With 20+ pages and dozens of interfaces, handwritten API functions and handwritten request and response types were too repetitive and too easy to drift out of sync.
Choice
Use @dandelion/swagger to generate request functions and TypeScript types from the backend Swagger docs, and sync them with one command.
Why
The moment the backend changed an API, the frontend could regenerate and immediately see type mismatches at compile time. That moved contract problems much earlier and also cut out a lot of repetitive interface-writing work.
Cost
We had to spend some time with the backend team making the Swagger docs disciplined enough to generate clean contracts. Once that was in place, it paid off for the rest of the project.
Decision 6 · Shared components: enough abstraction, not maximum abstraction
Problem
A back-office system always ends up with a shared component layer: search groups, filters, uploads, table actions, form fields, and so on. The question was how far to push that abstraction.
Choice
Only abstract patterns that were clearly high-frequency, and stop there. We did not try to force 100% coverage with “one universal component.”
Why
Over-abstracted admin components quickly turn into “universal but painful” monsters with too many props and too much hidden logic. We wanted shared components to stay focused on patterns like list search, action columns, and filter bars that were actually stable and reused.
Cost
Some lower-frequency scenarios still ended up with a bit of repeated page code. That was acceptable, because it was not the bottleneck.
