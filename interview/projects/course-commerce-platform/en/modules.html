05Core Modules

Five key modules, one by one

Each business area was its own module, but they all shared the same routing, permission, request, form, and component foundation. This section walks through them one by one.

01 · Project Center template/

This was the starting point for ops to run campaigns. One project meant one landing page plus all of its versions. It contained four sub-modules:

Existing projects
Course projects and navigation projects, with list pages, version management, copy, edit, preview, publish, and unpublish actions.
Template Center
Centralized management of course landing-page templates. Ops could create projects from templates instead of repeating the same setup work.
Component Library
Independent CRUD for 5 reusable component types such as popup, completion page, SMS, form, and button material, all reused in landing-page configuration.
Mini Program List
List-based management for mini-program related configuration.
02 · Gift Center present/

This module handled everything the business could “give” or sell. It broke down into three product types:

Course products
Course supply chain plus product center. It wrapped scattered course SKUs into real commercial product objects, with pricing, spec, inventory, and ownership.
Physical goods
Course boxes and physical boxes, meaning real shipped goods that connected to warehousing and fulfillment.
Virtual goods
Coupons, including creation, distribution rules, validity windows, usage restrictions, and batch management.
Deeply nested forms Batch operations Field linkage
03 · Tools tools/

This was one of the highest-frequency areas for daily ops work. “Small tools” only described the module size, not the business value.

Promotion tools
Generate, manage, and review campaign links and QR codes.
Custom campaigns
A configuration entry point for non-standard campaign scenarios, so one-off needs could still be absorbed into the platform instead of handled ad hoc.
Channel QR codes
A core attribution capability: one QR code could sit behind multiple customer-service accounts, with automatic distribution and attribution tracking.
Troubleshooting tools
A self-service panel for ops and customer support. Enter a user ID and quickly inspect orders, coupons, and attribution flow without pulling in engineering.
04 · User Portrait user-portrait/

This module turned “which users can buy which courses” into a configurable rule system. In practice, it was a frontend configuration layer for a rule engine.

User purchase limits
Rules based on user labels, identity, purchase history, and similar dimensions.
Course purchase limits
The same idea from the course side: who can buy this course, how many copies per user, and whether there is any tiering.
Course relationships
Relationships such as mutual exclusion, substitution, and upgrade paths between courses. For example, if a user bought A, can they still buy B?
Rule-engine frontend Dynamic Formily schema Condition-builder UI
05 · Workspace (low-code builder) work-space/

This was the entry point for ops to build course landing pages by themselves. The core capability was a 7-step wizard + Formily-driven page configuration + real-time mobile preview + save / preview / publish loop. It supported 8 landing-page templates, multiple delivery carriers like H5 and mini programs, plus version management and drafts.

Low-code page building is a separate highlight project in the portfolio, and that project goes much deeper and more complete. Here it only appears as one module inside the larger commerce admin, so I do not expand it further in this project.

A few design choices that mattered across modules

  • One permission model covered all modules — dozens of permission keys in access.ts drove menus, routes, and buttons together. A new module only needed to plug in one key to enter the whole system.
  • Shared components were reused across modules — form search groups, table action dropdowns, upload components, and Formily field extensions all behaved the same everywhere.
  • The API layer was type-generated — backend changes an interface, frontend regenerates Swagger types, and incompatibilities show up during compile instead of later.
  • Shared Redux slices handled cross-page state — config, user info, and troubleshooting state were shared globally instead of being passed through page trees.
