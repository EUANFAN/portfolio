07Challenges & Solutions

Five tough problems, and how I actually got them solved

Each challenge has the same shape: the problem, the architecture diagram, how I broke the solution down, a before/after code comparison, the key details, the result, how I'd say it in an interview, and the follow-up Q&A. The idea is that once you finish one section, you can walk into an interview and talk about it on your own.

01CRUD framework abstraction

20+ list pages, and we can't afford to rewrite filters, pagination, and request logic from scratch on every one of them.

The problem

The admin system had 20+ list pages — course projects, templates, widgets, coupons, course products, physical goods, promotion tools, channel QR codes, user purchase limits, course purchase limits, and so on. The page skeletons were almost identical:

  • A filter bar at the top: a group of search fields (name, status, date range, creator, and so on) plus search and reset buttons.
  • A table in the middle: columns, pagination, sorting, column show/hide.
  • An actions column on the right: view, edit, duplicate, delete, publish/unpublish, all gated by permissions.
  • A bulk actions bar at the top or bottom: check some rows and run an action across them.

If every page built all of that from scratch, it caused three real problems:

  1. Code size explodes. A hand-written list page easily hits 400–500 lines, and 80% of it is boilerplate.
  2. Behavior drifts. On page A, paging resets the filter. On page B, it doesn't. On page A, clicking the column header triggers a search. On page B, you have to hit a button. Those little differences pile up over time.
  3. Global changes are a nightmare. Something like "sync filter state into the URL" means touching 20+ files.

Core idea

Take everything every list page has to do anyway, and turn it into 3 hooks + 3 components. From that point on, a business page only cares about two things: 1) defining columns and 2) specifying the request function (searchFn). Filter state, pagination state, request timing, keeping filters across page changes, resetting current on a new search, loading state, column persistence — all handled in the primitive layer.

Code comparison
Before · hand-writing a coupon list page (the main boilerplate out of ~450 lines)
const CouponListBefore = () => {
  // filter form state
  const [params, setParams] = useState({ name: '', status: undefined })
  // list data
  const [dataSource, setDataSource] = useState([])
  const [loading, setLoading] = useState(false)
  // pagination state
  const [pagination, setPagination] = useState({ current: 1, pageSize: 20, total: 0 })

  // fetch function
  const fetchData = async (p = params, pg = pagination) => {
    setLoading(true)
    try {
      const res = await getCouponList({ ...p, page: pg.current, size: pg.pageSize })
      setDataSource(res.list)
      setPagination({ ...pg, total: res.total })
    } finally { setLoading(false) }
  }

  // initial load
  useEffect(() => { fetchData() }, [])

  // search: go back to page 1
  const handleSearch = () => fetchData(params, { ...pagination, current: 1 })
  // reset: clear filters + go back to page 1
  const handleReset = () => {
    const empty = { name: '', status: undefined }
    setParams(empty)
    fetchData(empty, { ...pagination, current: 1 })
  }
  // page change: keep filters
  const handleChange = (pg) => fetchData(params, pg)

  // ... plus 200+ lines of column persistence, bulk actions, permission buttons

  return (
    <>
      <Form>...search fields JSX...</Form>
      <Table dataSource={dataSource} pagination={pagination} loading={loading} onChange={handleChange} />
    </>
  )
}

Key details (likely follow-up points)
  • useTableSearch signature. Input is { searchFn, defaultParams, transform }, output is { tableProps, searchFormProps, refresh, params }. tableProps spreads straight onto antd Table, searchFormProps spreads onto the filter-bar component. The outer component doesn't have to know what state lives inside.
  • Unified pagination reset rules. Click search → current goes back to 1. Click a page number → filters are kept. Click reset → params clear and current goes to 1. This used to be written differently on every page. Now it's defined once and every list page behaves the same way.
  • Request timing. Triggered when defaultParams changes or when refresh() is called manually. Filter field edits do not auto-search (to avoid accidental fires). You have to hit the search button explicitly. This is a default we discussed — pages with a special need can opt out.
  • Column persistence key. The second argument of useTableSet(columns, 'coupon-list') is used as the localStorage key, so different list pages don't overwrite each other's column config.
  • Permission filtering in the actions column. TableOperationDropdown takes an actions array, each item can carry an authKey, and internally it uses useAuthCheck(authKey) to filter out items the user isn't allowed to see. Business code never writes permission checks by hand.

How I'd say it in an interview

"In an internal admin project, list pages are almost always the most repeated thing. Our project had 20+ of them, all with the same structure — filter on top, table in the middle, actions on the right, sometimes bulk actions. If every page built it from scratch, you'd get 400–500 lines per page, thousands of lines of boilerplate across the admin, and the paging and reset behavior would drift between pages over time."

"My approach was to pull out a CRUD primitive layer — two hooks, useTableSearch and useTableSet, plus three components: FormSearchGroup, FilterComponent, and TableOperationDropdown. After that, a business page only defines its columns and its request function. The primitive layer owns everything else."

"The result was that a new list page dropped from half a day to about two hours, and the same page went from 450 lines to around 80. The bigger win though was consistency — the user gets the same behavior on every list page in the admin. That's the part that pure code reduction doesn't capture."

Why not just use Ant Design Pro's ProTable directly?
ProTable is a fine starting point, and we do use it. But its API is pretty "big" and tightly coupled — search, pagination, and requests are all handled inside one component. When a page needs finer control — say, "don't reset filters on paging but do reset sorting," or "debounce this one field but nothing else" — you end up fighting ProTable's defaults. So we wrapped a thin layer of our own hooks and components around it. Simple list pages use ProTable directly, complex ones go through our wrapper. The two coexist.
How is useTableSearch implemented internally?
Basically four pieces of state — params, pagination, dataSource, loading — managed with useState, plus three event handlers — onSearch, onReset, onChange — and a useEffect to trigger searchFn at the right time. ahooks' useRequest would have worked too. We hand-wrote it because we wanted full control over request timing. The point wasn't which library we used — the point was to set the rules once and apply them everywhere.
What if a list page has a non-standard requirement, like drag-to-sort the whole table?
My rule is don't force non-standard requirements into the shared components. Something like drag-to-sort belongs in the one page that needs it. The shared components should only care about what most list pages do. If we grow the shared API to cover a 10% case, the 90% pays for it. Over-abstraction is the easiest trap to fall into in an internal admin project, so you really have to hold the line.
Why don't you put pagination and filter state in the URL? Won't users lose everything on refresh?
By default, no, and that was intentional for two reasons. One, URLs get ugly fast. Two, these admin pages don't really get shared around. For the few pages that actually need it — say, "send this filtered result to a colleague" — useTableSearch has a flag to sync params into URL search. "Simple by default, opt-in for special cases" is how I usually handle trade-offs like this.
How do you handle column-level permissions (not everyone can see a specific column)?
Two ways. One, each column definition can carry an authKey, and useTableSet filters it out at init time using useAuthCheck. Two, the page can conditionally render columns itself based on useAuthCheck. The first is more consistent, the second is more flexible. Both are supported.

02Fine-grained permission system

Three levels of access — menu, route, button — and 10+ roles. Scattered if/else checks would be a disaster.

The problem

The admin was used by 10+ roles at the same time: operations, marketing, product, support, team leads, admins, and so on. Each role saw a different menu, different pages, different buttons:

  • Menu level. Regular ops can't see the "System Management" menu. Product team can't see the "User Profile" module.
  • Route level. Ops can reach the "Coupon List" page but can't reach the "Coupon Configuration" detail page.
  • Button level. On the same page, regular ops can only view, the team lead can "approve," and the admin can "delete."

If we scattered permission checks through business code, we'd end up with dozens of places writing if (user.role === 'admin' || ...). The moment the rules changed, we'd have to find every one of them, and we'd miss a button somewhere. Guaranteed.

Core idea

"Data in one place, switches declared in one place, consumed in three places." All permission data is fetched once at startup. All switches are declared once in access.ts. Business code has three ways to consume — route, page, button — but they all read the same access object. Business code never touches raw permission data. It only touches named switches on access.

Key details
  • The type field convention. Each permission node has a type. type=0 and type=1 mean menu. Anything else (like type=2) means a function button. The frontend walks the tree in getInitialState and pushes each node into menus[] or functions[]. That's the contract we agreed on with backend.
  • initialState is fetched only once at startup. There's no live subscription. Permissions don't change often — a refresh or a re-login is enough. If that ever needs to change, we can just call Umi's setInitialState to refresh.
  • The frontend is not a security boundary. We made this explicit with backend: frontend gating is UI convenience — it hides entries the user can't use, and it avoids pointless requests. Real security is enforced by backend APIs. If a user tweaks the URL to bypass the frontend, the worst they can reach is the page shell. The data API will reject them.
  • useAuthCheck cost. It's basically Array.includes on a string array. Even with 1000 keys, we're talking microseconds. Not a performance concern.
  • Cost of onboarding a new module. Add a route in routes/index.ts, add a switch in access.ts. That's it. No other base code has to change.

How I'd say it in an interview

"The thing you really don't want in an admin system is scattered permission logic — dozens of files with if (user.role === 'xxx'). The moment the rules change, you have to hunt them all down, and you'll miss a button. In our case we had 10+ roles and three layers of access — menu, route, button — so scattering it would've been fatal."

"What I did was fetch once, declare once, consume in three places. At startup we pull the permission tree, split it by the type field into menus and functions, and drop it into Umi Max's initialState. Then in access.ts we declare all 50+ semantic switches — things like canSeePresentCoupon. After that, business code only touches named switches: the route config has an access field, the page wraps itself in <Access>, and buttons use useAuthCheck."

"The big win is that rule changes become a one-file edit (access.ts). Business code doesn't have any permission strings in it. A new teammate can open access.ts and immediately see every permission the system has and what it controls. And we were explicit with backend: the frontend is for UX, the backend is the real security boundary."

What if permissions need to take effect in real time — an admin revokes a permission and the user should immediately stop seeing the button?
Default is "refresh or re-login." If real-time matters, I'd layer in: (1) a long-poll or WebSocket notification when permissions change, (2) call Umi's setInitialState to refetch, (3) let access recompute, and useAccess picks it up automatically. In our project permissions weren't changing often, so we didn't build that — the simple version was enough.
How do you make sure nobody forgets to configure a permission when they add a page?
Two safety nets. One, code review on routes/index.ts enforces that every node has an access field — no access, no merge. Two, any path not in the permission tree falls through to a 403 page. Worst case, a user hits 403 — no one ever reaches a page they shouldn't see.
Isn't maintaining 50+ switches in access.ts painful?
Honestly, no, as long as it's grouped by business domain and commented well. We split it by project center / gift center / utilities, with comments at the top of each group. Adding a new switch is just copying a line and renaming it. That's still much cheaper than hunting hardcoded permission strings across hundreds of business files.
What's the difference between useAuthCheck and the <Access> component? When do you use which?
<Access> comes from Umi Max, takes a boolean from the access object, and is mostly for wrapping a whole page or section, usually with a fallback. useAuthCheck is our own hook, works against the functions array, and is for button / action-level checks where we want a boolean for conditional rendering. Coarse-grained: Access. Fine-grained: useAuthCheck.
Fetching permissions on startup is synchronous — is the first load slow?
Umi Max's getInitialState returns a Promise, and Umi shows a loading component (loading.tsx) until it resolves. The permission API takes 200–400ms in practice, so the user sees a short flash. Acceptable. If we wanted to make it faster: (1) HTTP caching + ETag on the permission endpoint, (2) write the result to localStorage and refresh it in the background after bootstrap. We're doing option 1 today.

03SSO login and unified request pipeline

How the token is fetched, where it's stored, what happens when it expires — business code shouldn't have to care about any of this.

The problem

We used the company's SSO for single sign-on across multiple sub-systems. The frontend had at least four things to handle:

  • Getting the session. First visit, or a jump from another sub-system that carries a token in the URL.
  • Token injection. Every request has to carry credentials.
  • Business error UX. Backend returns stat / code fields. On error, we show a message.
  • Expired token fallback. On 401, redirect to SSO logout with a redirect_url, so the user lands back on the same page after re-logging in.

Core idea

"Business code only deals with the happy path." Everything related to auth and error handling lives in the two axios interceptors. From the business side, an API call looks like a local function call: const user = await getUserInfo() — you get the typed UserInfo directly. No response-shape unwrapping, no error toasts, no token expiration logic in the business layer.

Code example
axios instance + request/response interceptors
// utils/axios.ts
const instance = axios.create({
  baseURL: config.host,
  timeout: 15000,
  withCredentials: true,
  headers: process.env.NODE_ENV !== 'production' ? { isDebug: 1 } : {},
})

// —— request interceptor: inject token
instance.interceptors.request.use((conf) => {
  conf.headers = { ...conf.headers, Authorization: getToken() }
  return conf
})

// —— response interceptor: unwrap + business error + 401 redirect to SSO
instance.interceptors.response.use(
  (response) => {
    // 401: token expired
    if (response.data?.code === 401) {
      message.error('Not logged in')
      return response.data.data
    }
    // business error (stat=1 means success by contract)
    if (response.data?.stat !== 1) {
      message.error(response.data?.msg || 'Internal error')
      return Promise.reject(response.data)
    }
    // success: unwrap data.data so business code gets the business object
    return response.data.data
  },
  (error) => {
    if (error?.response?.status === 401) {
      // token expired: clear token + redirect to SSO logout with redirect_url
      message.error({
        content: 'Your session has expired. Please sign in again.',
        onClose: () => {
          postLogout().then((res) => {
            clearToken()
            window.location.href = `${ssoLogout}?path=${encodeURIComponent(res.redirectUrl)}`
          })
        },
      })
    } else {
      message.error('System error')
      return Promise.reject(error)
    }
  },
)
Business-side call (no auth or error boilerplate)
// Types are auto-generated from Swagger
const Page = () => {
  const [user, setUser] = useState<UserInfo>()

  useEffect(() => {
    // one line — success gives us data, failure already toasted, 401 already redirected
    getUserInfo().then(setUser)
  }, [])

  return <div>{user?.name}</div>
}
Key details
  • Token in cookie, not localStorage. Cookies can be written cross-domain by the SSO service (with the right domain config) and backend can read them directly. localStorage needs frontend JS to run first, leaving a small gap. The trade-off is that our cookie isn't HttpOnly because the frontend has to read it — that's a known compromise.
  • Unwrapping response.data.data. The backend response always has the shape { stat, code, msg, data, traceid }. The interceptor returns response.data.data, so business code gets the actual business object directly. Combined with Swagger-generated types, API calls look like local function calls.
  • isDebug=1 in dev. A debug header we agreed on with backend. With it, backend skips parts of the login check so local dev is smooth. Production never sends this header, so there's no security risk.
  • 401 redirect_url round-trip. We use encodeURIComponent(window.location.href) to pass the current URL to SSO. After login, SSO brings the user back. URL query state (like filter params) is preserved, but in-memory form state is lost — which is a real pain point we could still improve by adding a localStorage draft.
  • Why code=401 isn't a reject. This is a backend convention: code=401 means "not logged in, but it's okay to silently return null." The business layer decides whether to show a message. It's different from network-level HTTP 401, which the outer catch handles by redirecting to SSO.

How I'd say it in an interview

"We used the company's SSO for single sign-on across multiple sub-systems. On the frontend side there were four things I had to handle: getting the session, injecting the token, showing consistent business error messages, and auto-redirecting on token expiry. If you write that at every call site, you end up with hundreds of repeated pieces of code."

"What I did was push all of that into the two axios interceptors. The request interceptor reads the token from cookie and injects Authorization. The response interceptor unwraps data.data so business code stays clean, handles the stat/code fields uniformly — business errors become toasts, 401 clears the token and redirects to SSO with a redirect_url so the user lands back on the same page."

"From the business side, an API call looks like a local function: const user = await getUserInfo(). You get a typed UserInfo back. No error handling, no token logic. And because Swagger auto-generates the types, the whole pipeline stays type-safe end to end."

What if the token expires while the user is filling in a long form? They lose their work on redirect.
That's a real pain point we discussed. Today, URL query state survives (the redirect_url carries the full URL) but in-memory form state doesn't. The natural next step is a localStorage draft for long forms. We actually did that on the workspace (page builder) side. Other places don't have it yet — it's on the list of things I'd still improve. If an interviewer asks, I'd just say that honestly.
What if business code wants to handle an error itself instead of letting the interceptor show a toast?
Edge case, but reasonable. Default behavior is: interceptor toasts and rejects, business just needs a .catch to handle it. For the rare "the user doesn't actually need to see this error" case, you can pass silent: true on the request config, and the interceptor skips the toast. Same pattern as before: simple by default, opt-out for the edge case.
Why not use axios's config.validateStatus to handle non-2xx responses?
validateStatus only looks at HTTP status. Our "business success" is response.data.stat === 1. HTTP can return 200 with a failed business state, so we have to handle it in the response interceptor. Network-level errors (HTTP 4xx/5xx) are handled by axios's error handler. The two layers are kept cleanly separate.
How does Swagger's generated types play with your unwrapping logic?
The generator knows the backend wraps everything in {stat, code, data, ...}, so it declares the API function's return type as data.data's type — not the wrapper type. That way const user = await getUserInfo() hands you UserInfo directly, perfectly aligned with what the interceptor returns.

04Complex nested forms

Purchase rules, course relations — deeply nested structures with heavy cross-field dependencies.

The problem

A few form types in the admin were dramatically more complex than the usual CRUD forms:

  • Purchase limit rules. Combine conditions across user type, grade, historical order count, purchase window, and so on. Rules have AND/OR relationships between them.
  • Course relations. Define exclusive / replacement / upgrade relationships between courses. One course can link to several relation chains.
  • Channel configuration. A channel has multiple support accounts, each with its own routing rules and attribution logic.

Writing these with raw AntD Form leads to:

  • Field-level dependencies implemented via useEffect + form.getFieldValue + form.setFieldValue — lots of side effects, hard to trace.
  • Dynamic add/remove of sub-items (like adding/removing a condition inside a rule) has to be hand-managed. Bugs are almost guaranteed.
  • Form shape is tightly coupled to business data shape. Wanting to move rule definitions to a backend config becomes a refactor.

Core idea

Use Formily's reaction model to turn "imperative useEffect wiring" into "declarative reactions". The schema says "when A changes, B's visibility / default value / options change like this." All dependency logic lives in the schema, not scattered across business components. On top of that, the schema itself can be fetched from backend, which lets us configure rules without shipping code.

      type: 'string',
      title: 'Type',
      'x-component': 'Select',
      enum: TYPE_OPTIONS,
    },
    subType: {
      type: 'string',
      title: 'Sub type',
      'x-component': 'Select',
      // reaction: depends on type, refetch options + clear when type changes
      'x-reactions': {
        dependencies: ['type'],
        fulfill: {
          run: '$self.setState({ loading: true })',
          state: {
            dataSource: '{{$deps[0] ? fetchSubOptions($deps[0]) : []}}',
            value: '{{$deps[0] !== $self.modified ? undefined : $self.value}}',
          },
        },
      },
    },
    detail: {
      type: 'string',
      title: 'Detail',
      'x-component': 'Input',
      // reaction: depends on subType, only visible when subType === advanced
      'x-reactions': {
        dependencies: ['subType'],
        fulfill: {
          state: { visible: '{{$deps[0] === "advanced"}}' },
        },
      },
    },
  },
}

const RuleFormAfter = () => (
  <FormProvider form={form}>
    <SchemaField schema={schema} />
  </FormProvider>
)
// All reactions live inside the schema. The whole thing can be pushed to a backend config.
Key details
  • What Reactions actually are. An observer pattern across fields. dependencies declares "what I care about," fulfill describes "what to do when it changes." Formily internally builds a dependency graph, so only the fields that actually depend on a change re-evaluate. Performance is good.
  • Registering custom form items. Under components/formily/ we built project-specific inputs (Cascader, MultipleSelect, and so on). After SchemaField.register, the schema can use them via 'x-component': 'Cascader'.
  • Dynamic schema. The schema is a plain JSON object — it can be fetched from an API. For example, the schema for purchase limit rules can come from a backend config center, and the frontend just renders it. New rules don't need a code change.
  • Validation. Formily ships the common validators (required / format / max / min / pattern) and supports custom function validators. We registered project-wide validators (phone, ID number, uniqueness checks) in utils/validator.ts.
  • Performance. Formily's reactivity is fine-grained. Changing one field only re-renders fields that depend on it. Our biggest form has 60+ fields and there's no perceptible lag.

How I'd say it in an interview

"A few forms in this admin are much more complex than the usual ones — purchase limit rules, course relations, channel config. Lots of nested structure, lots of cross-field behavior. Pick a value in field A, and field B's visibility and options both have to change. If you write that with raw AntD Form, you end up with a pile of useEffect + getFieldValue + setFieldValue. The dependency logic is scattered across the component and it's hard to maintain or test."

"I went with Formily and its reaction model. All the cross-field behavior goes into the schema as declarative reactions — what a field depends on and what it does when those change, all in one place. And because the schema is just JSON, we can push it to a backend config service, which means new rules can go live without a code release."

"We also built up a library of custom form items under components/formily/ — 8 of them, things like cascader, multi-select, step number, trim input. In the schema, 'x-component': 'Cascader' just works. With Formily's validation on top, we've got forms with 60+ fields that run smoothly and stay readable."

Formily has a real learning curve. How did you get the team onboard?
Honestly, yes, there's a curve. It took a new team member roughly a week to finish their first complex form. I did three things: (1) I wrote up a cheat-sheet doc of the high-frequency Formily patterns we used on the project, (2) I paired with a couple of teammates on their first few PRs, explaining reactions and lifecycle as we went, (3) when a tricky cross-field case came up, I'd take it myself and write the approach into the doc afterwards. After about two weeks, the team was comfortable.
Why not build your own schema-form system?
I considered it. The upside is "perfect fit for the business." The downside is that you have to implement reactions, validation, dynamic add/remove, and all the rest from scratch. Formily already does all of that well, and we didn't have needs unique enough to justify building our own. For an internal admin project, you have to resist the "build my own" instinct and spend the time on business and architecture instead.
What's the fundamental difference between Formily and AntD Form?
AntD Form is imperative — you call getFieldValue / setFieldValue and wire dependencies with useEffect. Formily is declarative and reactive — the schema declares dependencies, and Formily maintains the dependency graph internally, so changes propagate automatically. The real practical difference is where the dependency logic lives: AntD Form scatters it through business code, Formily keeps it in the schema.
How are the expression strings in x-reactions evaluated? Any security concerns?
Formily has its own expression evaluator. It's not eval. Expressions have access to $self, $deps, $form, and whatever custom scope functions you register. Because the evaluation is sandboxed, there's no arbitrary code execution.
How do you keep big forms fast? 60+ fields, no lag?
Formily's reactivity is fine-grained. Only fields that depend on a changed field re-render — not the whole form. Our largest form has 60+ fields and input latency is not noticeable. If we ever get to a few hundred fields, we can add <FormConsumer> for partial subscriptions or virtual scrolling inside <ArrayItems> to push it further.

05Multi-environment builds

Four environments — dev / mock / gray / prod — and they must never bleed into each other.

The problem

Four environments: dev (development), mock (mock data), gray (canary), prod (production). Each one has its own backend host, SSO URL, and logging endpoint. We could not have a dev accidentally start locally pointed at prod and write junk to it, and we could not have a canary build accidentally point at dev.

Architecture
Scripts (package.json)
npm run dev mode=dev
npm run dev-mock mode=mock
npm run gray mode=gray
npm run build mode=production
npm run build-gray mode=gray (build output)
Env injection (cross-env)
process.env.mode Injected by cross-env
hostEnum Centralized in src/config/index.ts
Runtime consumers
axios baseURL config.host
SSO endpoint config.sso
dev proxy target /proxyApi in .umirc.ts
Core idea

Explicit injection, centralized declaration, single point of runtime consumption. No reliance on defaults. No fuzzy NODE_ENV magic. cross-env mode=xxx forces the value in, src/config/index.ts owns the hostEnum with every environment's difference in one place, and business code only reads config.

Code example
// package.json scripts
{
  "dev": "cross-env mode=dev max dev",
  "dev-mock": "cross-env mode=mock max dev",
  "gray": "cross-env mode=gray max dev",
  "build": "cross-env mode=production max build",
  "build-dev": "cross-env mode=dev max build",
  "build-gray": "cross-env mode=gray max build",
}

// src/config/index.ts
export const hostEnum = {
  mock:       'http://localhost:3001/mock',
  dev:        'https://api-dev.xxx.com',
  gray:       'https://api-gray.xxx.com',
  production: 'https://api.xxx.com',
}

export default {
  host: hostEnum[process.env.mode as keyof typeof hostEnum] ?? hostEnum.dev,
}

// .umirc.ts — dev proxy uses the same hostEnum
const proxy = {
  '/proxyApi': {
    target: process.env.mode ? hostEnum[process.env.mode as keyof typeof hostEnum] : '',
    changeOrigin: true,
    rewrite: (path) => path.replace(/^\/proxyApi/, ''),
  },
}
Key details
  • cross-env for cross-platform env injection. Windows, macOS, and Linux each have their own env-var syntax. cross-env smooths that out. We had Windows users on the team, so this mattered.
  • CI scripts are self-documenting. npm run build-gray vs npm run build — you can tell at a glance which environment you're shipping to. No "am I sure I'm building the right thing?"
  • Business code is environment-agnostic. Need the API host? import config from '@/config'. Need to know if it's dev? process.env.NODE_ENV. No environment if/else scattered around.
How I'd say it in an interview

"Multi-environment build is something I intentionally kept simple. cross-env injects the mode, all environment differences live in src/config/hostEnum, and business code only reads config.host. The nice part is that the release script tells you exactly which environment you're building, local dev never accidentally points to prod, and adding a new environment is a two-line change."

How is the mock environment actually wired up?
We used Umi's built-in mock feature. Mock handlers go under mock/, and npm run dev-mock tells Umi to intercept matching requests and return mock data. It plays nicely with Umi's routing and needs no setup. We didn't pull in something like MSW — that's better suited for building a standalone component library, not an internal admin.
When something breaks on the gray environment, how do you locate it quickly?
Gray requests carry an X-Env: gray header so backend logs can filter by it. Frontend log reporting also includes mode. When something breaks, we use our internal "inspection tool" (check-tool module) — plug in a user ID, pull their recent request trace, filter by environment. That's way faster than digging through production logs.
