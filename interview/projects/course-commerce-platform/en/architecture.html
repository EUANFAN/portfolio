Key flow 1: permission flow (menu + button auth)

This was one of the most recognizable parts of the project: the backend permission list directly drove routes, menus, and buttons on the frontend.

SSO login → getInitialState() → Fetch {menus, permissions} → Split by type → Generate booleans in access.ts → Route guard + menu render + button auth

In the permission list, type=0/1 meant menu permissions, which drove route access and ProLayout menus. Everything else meant function permissions, which drove buttons. Pages used <Access accessible={access.canSeeXxx}>, while buttons used helpers like useAuthCheck('project-center-project-list-course-edit'). One key controlled route access, menu visibility, and button visibility together, so new modules only needed a route plus an access key.

Key flow 2: request and contract flow

The frontend/backend contract was generated from Swagger, so business code could stay focused on the happy path.

@dandelion/swagger → Generate TS types + request functions → Business code calls them → Request interceptor (SSO token / mode) → Backend → Response interceptor (401 → SSO · business-code toast)

@dandelion/swagger generated TypeScript types and request functions directly from OpenAPI, so the frontend almost stopped writing the API layer by hand. The axios interceptors centralized two important behaviors: 401 automatically logged the user out and redirected to SSO, and business error codes showed unified toast messages. That kept error-handling conventions inside the interceptor layer instead of leaking into every page.

Key flow 3: CRUD list-page flow (hook-driven)

More than 20 list pages shared the same behavior, and this was the abstraction that cut a page from 500+ lines down to around 150.

Declare columns + search form + fetchFn → useTableSearch → useAntdTable data source → useTableSet → ProTable render + localStorage persistence

useTableSearch took over search / reset / paging / URL sync, and returned helpers like { search, reset, formValue }. Search automatically reset current, paging preserved filters, and reset cleared params. useTableSet took care of custom columns, including width, visibility, and operation columns, while user changes were persisted into localStorage. The default behavior lived inside the hooks, so every new list page got the same interaction model automatically.
