Project Deep-Dive

Course Commerce Platform

A B-side operations platform for the internal ops team. It pulled all the scattered “make money” workflows in the course business into one admin system: product setup, marketing tools, user rules, and landing-page creation. Under the hood, it was built on centralized routing in Umi Max + a two-layer permission model, reusable CRUD and form abstractions, and Swagger-generated API contracts, so 5 business domains and 20+ pages could still share one interaction model, one permission model, and one data contract.

01Overview

What this project was, in one sentence

A commercial operations platform for the internal ops team, built to bring all the revenue-related workflows in the course business into one admin: product setup for courses, boxes, and coupons; marketing tools like promotion links, channel codes, and custom campaigns; user rules like purchase limits and user portrait rules; and finally landing-page setup and publishing. On the frontend side, we used Umi Max routing plus the access plugin for permissions, Redux Toolkit for global state, Formily for complex forms, ahooks for async and interaction flows, and Swagger-generated API contracts so frontend and backend stayed aligned on both types and semantics.

Business domains

Project Center
Course projects, navigation projects, template center, component library, and mini programs. This was the entry point for landing-page work.
Gift Center
Course products, physical goods, and virtual goods like coupons.
Tools
Promotion tools, custom campaigns, channel QR codes, and troubleshooting tools. Basically the everyday toolkit for marketing ops.
User Portrait
User purchase limits, course purchase limits, and course relationships. This is where we controlled who could buy what.
Workspace
The landing-page builder, built around a 7-step wizard, Formily forms, and real-device preview.
Situation
The operational workflow for the course business was spread across several old admin systems. Products were in one system, coupons in another, landing pages were still hand-built by frontend engineers, and purchase-limit rules required backend code changes. Every campaign meant hopping across 3-4 systems, while frontend was stuck patching one-off workflows with no consistent interaction model or data contract.
Task
Pull those scattered capabilities into one admin, give the business a consistent interaction model, a consistent permission model, and a consistent data contract, and let non-engineering ops teammates handle most of the daily workflow on their own. The bigger goal was to get the frontend team out of repetitive delivery work.
Action
Built the frontend foundation on top of Umi Max. We used centralized routes plus the access plugin to support menu-level and button-level permissions. We split the project into three layers: business modules, platform capability, and infrastructure. We abstracted common CRUD patterns into hooks like useTableSearch and useTableSet, used Redux Toolkit for global state, Formily for deeply nested forms, ahooks for async interaction flows, and @dandelion/swagger for generated API contracts. Axios interceptors handled auth, errors, and toast behavior in one place.
Result
One admin ended up covering 5 business domains and 20+ secondary pages. Ops no longer had to keep switching across old systems. Landing pages moved from “wait for frontend scheduling” to “ops can handle it,” and the frontend team shifted from “build pages” to “build the platform.”

Four things that really made it work

  • Centralized routing plus a two-layer permission model. One routes.ts declared all routes, and every node carried an access key. Umi Max then used that in one place to drive menu rendering, route blocking, and button-level auth. Pages used <Access>, buttons used useAuthCheck(authKey), and menu permissions stayed clearly separated from function permissions.
  • CRUD scenarios were abstracted into hooks instead of copied page by page. More than 20 list pages only needed to define columns and searchFn. Search, paging, sorting, column persistence, and URL sync all came from useTableSearch and useTableSet. A new list page went from 500+ lines down to roughly 150, and the behavior stayed consistent.
  • Formily carried the complex nested forms. Coupons, products, and rules all had deep nesting, field linkage, and heavy validation. Formily let us describe those as schemas and reactions instead of writing a mess of onChange and conditional state code by hand.
  • Swagger generation plus axios interceptors kept the contract layer clean. @dandelion/swagger generated TypeScript types and request functions from OpenAPI, so frontend barely wrote API wrappers by hand. Axios interceptors took care of 401 → SSO redirect, unified business-error toasts, and header injection, so business code mostly stayed on the happy path.

A few details that were easy to miss, but mattered a lot

  • The permission model was explicitly split. In the backend permission list, type=0/1 meant menu permissions and everything else meant function permissions. Menu permissions drove access and routing; function permissions drove buttons. That boundary was clear right at the frontend entry point, so the logic did not get tangled later.
  • One codebase, four build outputs. We used cross-env mode=dev|mock|gray|prod to run four environments. Mock let frontend work independently, gray handled staged rollout, and prod was the real release build. Same source, different outputs, no extra branches.
  • List-page behavior was standardized by default. Search reset current, paging preserved filters, reset cleared params, and column order and visibility were persisted in localStorage. All of that lived inside the hooks, so every new list page got the same default behavior for free.

Note: the workspace module in this project also included a low-code landing-page builder, but the team had a separate low-code platform that was much deeper and is presented as its own highlight project (Low-Code Campaign Platform). Here it is treated as one module inside the larger commerce admin, not the main story.
