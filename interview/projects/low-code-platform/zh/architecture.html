关键链路 1：从"拖一个组件"到"画布更新"

运营在左侧组件库点了一下「图片」组件，到画布出现这个组件的全过程——完全走 MobX 的响应式路径，不经过任何事件总线：

点击 / 拖入组件卡片 → Stage.addChild(widget) → widget push 到 @observable widgets 数组 → @observer 画布检测到 widgets 变化 → 自动 re-render 新组件

同时，右侧属性面板修改 widget.data.url 时也走同一条路径。 三栏（组件库 / 画布 / 属性面板）都是 @observer，共享同一个 StageStore——改哪里，哪里自动重绘，不需要事件传递。

关键链路 2：从"点发布"到"C 端秒开"

这条是整个项目最复杂也最要紧的链路：

运营点「发布」 → 前端 POST schema 到 Koa → Node 侧 jsdom 模拟浏览器 → 从 CDN 拉外部组件 JS · new Function() 注册 → require App · renderToString(<App />) → Handlebars 装配 HTML + PageData → HTML / JSON 传 CDN + 截图 → C 端拿 HTML 秒开 · hydrate 补交互

任何一步失败（组件 JS 下载出错、SSR 里调了浏览器专属 API、require 抛错……）， 就把 needHydrate 置 false，C 端退回 ReactDOM.render。 SSR 是锦上添花，客户端渲染是兜底——发布流程不能因为 SSR 挂掉。

另外一个容易被忽略的细节：每次发布都会清 App 模块的 require.cache， 防止上一次发布的旧组件代码污染这一次的渲染。jsdom 环境在 Node 里是有状态的，不主动隔离会出 bug。

关键链路 3：从"组件升级"到"历史活动页不被打穿"

组件开发者发新版本 → CDN 存 /components/{type}/{version}/index.js → 旧 schema 里的 version 保持不变 → C 端按 schema 钉死的版本拉 JS → 历史活动页继续使用旧版本

关键点是"版本钉在 schema 里"，不是"全站用最新版"。新活动页想用新版本，就在编辑器里升； 老活动页不动 schema，就永远停在它发布时的那个版本——组件升级不会回灌到历史活动。
