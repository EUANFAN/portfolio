团队与组织影响

  • 前端团队职能转型。从"交付型"（每天写活动页）转成"平台型"（维护编辑器、组件库、SSR 引擎）。平台工作开始能排进迭代计划。
  • 运营自助率大幅提升。运营不再依赖前端排期——90%+ 的活动从"提需求给前端"变成"自己配发布"。
  • 第三方团队参与组件生态。几个业务前端团队主动开发组件上架，形成了内部的"组件应用市场"。
  • 公司内部复用。平台的 schema 格式和渲染引擎被其他团队复用到了"内部门户页搭建"等场景。
  • 知识沉淀。整个流水线、组件契约、x-cli 脚手架都有文档和视频教程，新人两天能独立写组件。

我个人的成长

  • 完整走完"平台型项目"的生命周期。 从"识别业务瓶颈 → 拍架构 → 推落地 → 治理迭代"——四个阶段都亲自经历过，才真正理解为什么说"做平台最难的不是技术"。
  • 在"响应式状态"、"SSR / jsdom"、"动态模块"三个方向都留下了判断。 这些不是看文章能学到的——都是在线上 bug 里一点点摸出来的。
  • 从"能写复杂前端"到"能主导平台型项目"。 最大的变化是看事情的角度：不再是"这段代码怎么写"，而是"这个能力给谁用、能用多久、怎么演进不崩"。
  • 推动非技术事情的能力。 低代码平台一半的挑战是技术、另一半是推广。运营培训、产品沟通、业务方 PK——这些事以前我会躲，现在会主动接。
  • 对"克制"的理解。 低代码最容易膨胀——什么都想做、什么都想支持。我学到的是"先把边界画死，不做什么比做什么更决定平台能不能活"。 坚持不开代码面板、坚持三层组件分级、坚持所见即所得——这些"不做"的决定，反而是平台长到今天的关键。

如果重做一遍，我会不一样的地方

  • 更早引入 TypeScript。项目用了 JS + jsconfig，对重构不够友好。TS 的类型能力在 schema 驱动的场景里其实非常值钱。
  • 组件 SSR 兼容的检测前移。我们是 SSR 上线后才加的 xcli ssr-check，之前踩了很多"组件里用 window"的坑。工具应该一开始就提供。
  • 早点做组件市场 UI。前两年外部组件是通过内部文档索引的，后来才做可视化的市场，用户发现新组件的成本长期偏高。
  • 更早用 Playwright 做端到端回归。编辑器的交互很多，手动测试 cover 不全，上了 Playwright 之后才有信心做大改。
