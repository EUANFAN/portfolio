05重点模块

5 个重点模块，逐一拆解

平台看起来是一个大产品，但技术含量不是平均分布的。下面这 5 块是投入最多、也最能体现架构判断的地方—— 分别是编辑器三栏模型、画布架构、组件体系、触发器、SSR 发布链路。

01 · 编辑器三栏模型 MobX 驱动的单一数据源

编辑器长什么样其实大家都见过——左边组件库、中间画布、右边属性面板。真正的技术含量在三栏怎么通信： 不是事件，不是 Redux，而是共享同一个 MobX StageStore。

数据模型
StageStore → Project → Pages → Widgets → Layers / Triggers / Animations，全链路 @observable，每个节点的属性改动都会被 MobX 追踪。
通信方式
三个面板都是 @observer。右侧改 widget.data.url，画布里的 <WidgetContent> 自动 re-render——没有 dispatch、没有 subscribe、没有 props 穿透。
一个绕不开的坑
MobX 4 的 observable({}) 不追踪动态新增的属性。所以 widget.modifyData(k, v) 里要写成 this.data = { ...this.data, [k]: v }，强制引用换新。这个细节我们踩过坑，后面抽到 Widget 基类里兜底。
Stage 抽象层
Project / Page / Layer / Widget 四种容器形状不同，但「加子节点」「删子节点」「选中子节点」「排序子节点」都一样。用 Stage 类做多态包装，编辑器里从此没有按容器类型分支的 if/else。
@observable @observer 单一数据源 多态抽象
02 · 画布架构 瀑布流 + 自由定位 并存

运营搭页面时有两种心智——「按顺序往下铺一列组件」和「随便放哪儿都行」。我们没有强选一个，而是两种布局共存于同一张画布。

瀑布流布局
用 react-sortable-hoc，组件按文档流从上到下排列，可拖拽排序。适合课程卡片、图文列表这种"结构感强"的模块。
自由定位布局
用 react-rnd，组件绝对定位，可拖拽移动 + 八向缩放，拖动步长 5px。适合"按设计稿精确摆放"的场景，比如浮层按钮、装饰元素。
多层级视口
SwipeableViews 包住画布，运营可以从"项目" → "页面" → "容器" → "图层"一路深入编辑，像点进文件夹，退出来就回到上一层。
画布周边
水平 / 垂直两把 CanvasRuler（像素标尺 + 对齐辅助线），一个 Selector（鼠标框选多个组件），让画布从"一个 div"变成真正可用的工作空间。
react-rnd react-sortable-hoc SwipeableViews CanvasRuler · Selector
03 · 可插拔组件体系 内建 / 内部 / 外部 三层

组件是这个平台最活跃的资产。分三层的目的是平衡"稳定"和"灵活"——越稳定的打包进客户端，越业务化的从 CDN 运行时拉。

内建组件
Image / RichText / NormalText / Container / DataContainer / Layer——所有业务都会用的底层积木，构建时直接 import 进客户端 bundle。
内部组件（PageData.innerWidgets）
业务定制组件（比如"课程卡片"），每个独立打 bundle 传 CDN，运行时 loadScript 动态加载。发更新版本不会打穿客户端。
外部组件（PageData.installed）
第三方团队用 x-cli 脚手架开发、提交审核、上传 CDN。版本号写在 schema 里——C 端按 schema 钉住的版本加载，历史页永远不会被升级打穿。
统一契约
不管哪一层，组件定义长一个样：{ name, type, data, config, methods, onRender, onEnter, onNext }。config 直接被右侧属性面板消费生成表单，写组件 = 写组件定义。
组件注册中心
所有组件最终汇聚到 WidgetConfigs 这个 @observable 数组，作为全局注册表。外部组件通过 god.__components__ 自注册，跟内建组件统一视图。
动态 import loadScript · loadStyle semver 钉死版本 x-cli 脚手架
04 · 触发器体系 用声明式替代"让运营写代码"

触发器是我们坚持「不让运营写 JS」的关键。所有交互都用声明式描述——什么事件，触发什么 action。

事件类型
onClick / onShow / onScroll / onCustomEvent——覆盖活动页 95% 以上的交互场景。
Action 集合
Redirect（跳转）· ScrollToPage（滚动到指定页面）· ChangeWidget（修改其他组件状态，实现跨组件通信）· EmitListeners（发自定义事件，配合 onCustomEvent 实现订阅 / 发布）。
为什么不开代码面板
一旦允许运营写 JS，审计 / 安全 / 排障就全失控。声明式触发器虽然有上限，但覆盖了绝大多数场景，把平台的稳定性承诺守住了。
可视化调试
编辑器里能看到触发器的挂载关系——谁触发了谁、按什么顺序执行。触发器一多容易成迷宫，可视化这一刀很关键。
声明式 跨组件通信 埋点 hook 自动挂
05 · SSR 发布流水线 Koa + jsdom + renderToString + 降级

这条链路是整个项目里技术密度最高的地方，细节多、失败面大、但一旦跑通，用户就能享受 1s 级首屏。

环境模拟
jsdom 在 Node 里创建一个"假浏览器"——document / navigator / localStorage 都得有，否则第三方组件一调浏览器 API 就炸。
组件预加载
按 schema 里的 installed 列表从 CDN 拉所有外部组件 JS，用 new Function() 执行，让它们注册到 god.__components__。只有带合法 xcli 元数据的组件才允许服务端执行——安全兜底。
React SSR
require 客户端的 App，renderToString(<App />) 生成 mainHTML。Handlebars 模板把 mainHTML 和 PageData（JSON）嵌进最终 HTML，传 CDN。
require.cache 隔离
每次发布都清 App 模块的 require.cache，不然上一次发布的旧组件代码会污染这一次渲染。这个细节不搞会出诡异 bug。
优雅降级
只要 SSR 任何一步失败，就把 needHydrate 置 false，C 端退回 ReactDOM.render。用户会慢一点看到页面，但绝不会看到白屏——发布流程 SLA 远比 SSR 收益重要。
C 端 hydrate
浏览器拿到 SSR HTML 后，ReactDOM.hydrate 在已有 DOM 上绑事件监听——不重建节点，不闪屏，不掉 LCP。
jsdom renderToString Handlebars hydrate require.cache 隔离 优雅降级
