项目深度梳理

企业级低代码活动搭建平台

一个面向运营、市场、产品团队的可视化搭建平台：把 H5 活动页的生产，从「每个活动排前端工期」，变成「运营自己拖一拖、配一配、发布」。 底层是 MobX 驱动的三栏编辑器 + 可插拔的组件体系 + Node 侧 SSR 发布流水线，把所见即所得和秒开两件事一起做了。

01项目概览

一句话讲清楚这个项目

运营在三栏编辑器（左侧组件库 / 中间画布 / 右侧属性面板）里拖组件、调样式、挂触发器， 画布所见即所得，点「发布」后 Node 侧会用 jsdom + renderToString 把页面 SSR 成一份可以秒开的 HTML， 传到 CDN；C 端访问时先拿到 SSR 的 HTML（首屏直接出画面），再 ReactDOM.hydrate 绑事件，整个活动页就变成可交互状态。

业务域一览

搭建与编辑器
三栏编辑器（组件库 / 画布 / 属性面板）、拖拽编排、对齐与层级、基础容器与自由布局。运营日常工作的主入口。
物料与模板
活动页模板、区块复用、页面配置（主题色 / 埋点 / 分享信息 / 跳转策略）、跨活动的标准化“搭建底盘”。
组件生态
三层可插拔组件（内建 / 内部 CDN / 第三方 x-cli）、版本治理、运行时动态加载与注册，保证“能扩展、可回滚”。
发布与运行时
Node 侧 SSR 发布流水线（jsdom + renderToString + Handlebars）、CDN 分发、C 端 hydrate、失败优雅降级与保底发布。
数据与治理
页面数据模型（schema / PageData）、触发器与埋点约定、组件升级策略、线上稳定性兜底（如 require.cache 隔离）。
背景（S）
活动节奏从「月」加快到「周甚至天」，但每个活动页都得前端从零写一份——组件库各写各的，样式兜底各写各的，埋点也各写各的。前端一半人力被套在重复交付里，平台型工作常年推不动。
目标（T）
把前端从活动交付链路里拿掉：运营能自己配、自己预览、自己发布；同时保证编辑器上看到的和线上看到的一模一样，首屏不能慢，线上不能因为发布失败挂掉。
动作（A）
用 MobX 把编辑器的三栏数据统一进一个 StageStore，改任何一个地方，另外两个地方自动联动； 组件做成三层可插拔（内建 / 内部 CDN / 第三方 x-cli）+ 运行时动态加载； Node 侧搭 SSR 发布流水线：jsdom 模拟浏览器环境，把 App 整棵树 renderToString 出来，Handlebars 装进模板传 CDN； 同时做优雅降级——SSR 任何一步失败，客户端就退回 ReactDOM.render，发布流程保底不中断。
结果（R）
活动页上线从「天」级压到「十几分钟」级；前端从「交付型」变成「平台型」，接下来的精力全都投入编辑器、组件生态和 SSR 性能；累计承载了公司内部几条业务线的几乎全部活动页。

让它真正跑起来的四个关键点

  • MobX 驱动的三栏响应式。 左（组件库）、中（画布）、右（属性面板）三个 @observer 组件共享同一个 StageStore。 属性面板改 widget.data.url，画布自动重绘，不用事件总线，不用 Redux 样板代码。
  • 可插拔组件体系（内建 / 内部 / 外部三层）。 内建组件（Image / RichText / Container 等）打包进客户端；内部和外部组件按版本从 CDN 动态 loadScript； 每个组件定义里有 data（默认数据）、config（声明式属性面板）、onRender（共享渲染函数）、methods。 第三方开发者用 x-cli 脚手架就能发组件，不用动平台代码。
  • 所见即所得的双端一致性。 编辑器的 BaseWidget 和 C 端的 Widget 都继承 WidgetClass，共用 onRender 和同一份 .less。 编辑器用 375px 固定画布，C 端用 px2rem（1rem = 37.5px）自适应，编辑器看到啥、线上就是啥。
  • SSR + 秒开 + 优雅降级。 服务端用 jsdom 模拟 document/navigator，从 CDN 拉外部组件的 JS、new Function() 执行注册， 再 renderToString(<App />)，Handlebars 把 HTML 和 PageData 缝到模板里传 CDN。 只要任何一步失败，就把 needHydrate 置 false，让客户端退回 render，发布绝对不能因为 SSR 挂掉。

几个容易被忽略、但其实很要紧的细节

  • MobX 4 observable({}) 不追踪动态属性——所以 widget.data 改值时必须整块重新赋值（this.data = { ...this.data, [k]: v }），不然下游面板不会刷新。
  • 每次 SSR 都得清 require.cache，不然上次发布的组件代码会污染这一次的渲染。
  • Stage 抽象层把 Project / Page / Layer / Widget 四种容器的 addChild / removeChildren / sortChildren 做成了同一个多态接口——编辑器代码里几乎不再出现「按容器类型分支 if/else」。
