06关键决策

几个关键节点上的选型与取舍

每个决策按「问题 / 选择 / 理由 / 代价」四段式讲。面试里讲清楚后两段，比讲清楚选了什么更重要—— 因为选什么体现知识面，讲理由和代价体现判断力。

决策 1 · 状态管理 —— 选 MobX，不选 Redux
面对的问题
编辑器里三栏（组件库 / 画布 / 属性面板）要高度实时联动。改一个属性，画布要立即重绘；选中一个组件，右边面板要立即切换。状态管理方案怎么选。
最终选择
MobX——一个 StageStore 包住 Project → Pages → Widgets → Layers 的所有数据，全部 @observable；三栏面板全是 @observer。
主要理由
编辑器里的状态嵌套深、引用多、改动频繁。用 Redux 的话要写海量的 action / reducer， 改一个字段要穿透好几层 reducer，而且对象嵌套下 immer 也帮不到多少。 MobX 的响应式模型正好对得上——改数据就是改对象，不用管派发，下游自动重绘。代码量预计能降到 Redux 的 1/3 以下。
代价
① MobX 的响应式有"魔法"，新人不了解会踩坑（比如动态属性不追踪）； ② 调试没有 Redux DevTools 那么直观，时间旅行要自己搭。 这两点我们都做了兜底——Widget 基类里把"动态属性"统一重新赋值；撤销重做单独维护一个 history 栈。
决策 2 · 组件形式 —— 选 Class Components，不追 Hooks
面对的问题
项目立项时 React Hooks 已经稳定一段时间了，但 MobX 4 对 Hooks 的支持不如 Class（@observer 装饰器 vs. observer() 函数式包装）。选哪个？
最终选择
Class Components——核心编辑器组件和 Widget 基类全部用 Class，装饰器风格写 @observer / @observable。
主要理由
三件事：①继承——WidgetClass、BaseWidget、Widget 三层继承关系，用 Class 天然表达，用 Hooks 要绕一大圈组合； ②装饰器语法——跟 MobX 的 @observable 天然配合，代码噪音比 observer(function) 少一半； ③生命周期显式——编辑器里大量依赖 componentDidMount / componentDidUpdate 做细粒度的时机控制，用 Class 比 useEffect 的闭包心智更直观。
代价
跟社区潮流不一致，新同学上手要适应 Class 写法；长期看 React 官方 Hooks 是主推方向，未来升级会有迁移成本。我们接受这个代价，以换取当期的开发速度。
决策 3 · SSR —— 自己搭 jsdom + renderToString，不用 Next.js
面对的问题
C 端首屏是硬指标。用 Next.js 这种成熟框架，还是自己基于 Koa + jsdom 手撸？
最终选择
自己搭——Koa 作为后端主框架，jsdom 模拟浏览器环境，ReactDOM.renderToString 产出 HTML，Handlebars 装配进模板。
主要理由
Next.js 适合"一个应用就是一个站点"的场景，但我们的场景是"一个发布服务给成百上千个活动页打 SSR"—— 每个活动的组件不同、版本不同、schema 不同，SSR 过程里还要按需从 CDN 动态拉第三方组件 JS 执行。 这种"多租户 + 组件动态注入"模型塞进 Next.js 非常别扭，倒不如直接基于 Koa + jsdom 手控每一步。 另外我们已经有自己的 Koa 后端，再引 Next.js 会让技术栈复杂度翻倍。
代价
自己手搭意味着 jsdom 边界问题、require.cache 污染、SSR 失败降级、外部组件沙箱——每一个都得自己搞定。 我们踩了大半年坑才把流水线跑稳。但换来的是对整条链路的完全掌控，出问题能查到毫秒级。
决策 4 · 组件体系 —— 三层分级，不统一用一种加载方式
面对的问题
组件是要全部打包进客户端，还是全部运行时从 CDN 拉？
最终选择
三层分级：内建打包 · 内部 CDN 动态加载 · 外部第三方 x-cli 独立发布到 CDN。
主要理由
全部打包进客户端——包体会随组件数量线性膨胀，几百个组件后首屏就崩了； 全部运行时拉——最基础的 Image / Container 都要等 CDN 回来才能渲染，首屏体验差。 按"稳定度"分三层是折中解：业务一定会用的（Image、Container）打进去，业务专属但稳定的（课程卡片）内部 CDN，第三方的外部 CDN。 用户体验和可扩展性都能照顾到。
代价
加载策略分三层，调试和排障要搞清楚组件在哪一层；"一个组件从内部升级到内建"需要一次发版。这些代价可控。
决策 5 · 运营交互能力 —— 坚持声明式触发器，不开代码面板
面对的问题
运营偶尔会有"非标"交互需求。最直接的方案是开个代码面板让他们写 JS，最省事。要不要开？
最终选择
坚决不开——所有交互必须通过声明式触发器（事件 + action）表达。非标需求走"抽象成新触发器"的通道，不走"让运营写 JS"。
主要理由
一旦允许运营写 JS： ①安全——脚本会在 C 端运行，XSS 风险无法收敛； ②审计——发布流程失去可预测性； ③排障——线上出问题，责任边界模糊（不是平台 bug 就是运营代码，扯皮）； ④演进——后面想升级 React 版本时，运营写的代码会到处炸。 声明式触发器虽然有上限，但能抽象出 95% 的实际需求，把平台的可预测性守住了。
代价
少数非标需求没法立刻满足，需要等我们在 action 体系里加新类型——响应慢一点，但换来的是长期可维护。
决策 6 · 双端一致 —— 编辑器 375 固定画布 + C 端 px2rem 自适应
面对的问题
活动页要在各种宽度的手机上都看起来协调。运营在编辑器里按什么宽度搭？C 端怎么适配？两端怎么保证一致？
最终选择
编辑器画布固定 375px（iPhone 基准宽度，所有 px 单位 1:1）； C 端上线时自动 px2rem 转换（1rem = 37.5px，根据视口宽度按比例缩放）。 编辑器 BaseWidget 和 C 端 Widget 共享 WidgetClass、共用 onRender、共用 .less——一份代码两端跑。
主要理由
运营不懂响应式，只能给他们一个确定的画布宽度——所见即所得的前提是"所见"本身是确定的。 选 375 是因为设计稿基本都按 iPhone 出。上线后 px2rem 把所有 px 值折算成 rem，视口宽度决定 rem 大小，全页面等比缩放—— 视觉比例永远不会错位。
代价
① 小尺寸手机字变得小，大屏幕字变得大，极端宽度（比如 iPad）会失真——但我们业务 99% 是手机，不考虑 iPad 投放； ② 组件开发者必须只写 px、不写媒体查询——这条写进规范，组件接入时卡住。
