03我的职责

我在这个项目里做了什么

前端负责人 + 核心贡献者，带一个 3–5 人的小团队。 编辑器、组件体系、Node 侧 SSR 发布流水线三条核心链路我都直接下场写过 —— 前端和后端（Koa / jsdom / MongoDB）的工作我都参与， 因为这类平台的关键决策只看一面都会走偏，得前后端一起打通才站得住。

整体技术架构

  • 关键技术选型。 选 MobX 而不是 Redux、保留 React Class Components、两种布局（瀑布流 + 自由定位）并存、 SSR + 客户端 hydrate、组件运行时从 CDN 动态加载、三层组件体系（内建 / 内部 / 外部）—— 这几个决定项目走向的选择都是我主导的，后面也都要为它们兜底。
  • 组件开发规范与契约。 Widget 基类（WidgetClass）、编辑器 / C 端双端包装（BaseWidget / Widget）、 组件的 data / config / onRender 四段式契约、x-cli 脚手架、 版本治理规则 —— 第三方同学用这套开发组件的规范是我定义并推动的。
  • 团队工程规范。 项目脚手架、ESLint / Prettier / Husky、Webpack 多环境配置（dev / gray / prod）、Docker 本地化部署、内部文档。

核心模块开发

  • 编辑器（前端）。 三栏通信模型、StageStore 数据结构、Stage 抽象层、画布工作区（标尺 / 视口 / 框选器）、 右侧 ControlWrap 动态表单渲染器 —— 这套核心骨架的第一版是我写的，后面逐步和团队一起完善。
  • 组件体系（前端）。 内建组件（Image / RichText / Container / FixedButton 等）的基础集、组件运行时的动态 loadScript 加载器、 版本路由（同组件不同版本并存）、编辑器和 C 端共享 onRender 的渲染机制。
  • Node 侧 SSR 发布流水线（前端 + 服务端都做）。 jsdom 环境准备、外部组件从 CDN 拉取并 new Function() 注册、renderToString、 Handlebars 模板装配、CDN 部署、require.cache 隔离、优雅降级策略 —— 前端侧和服务端侧（Koa 路由、发布接口、MongoDB schema 存储、资源上传）我都直接下场写， SSR 本身就是横跨前后端的事，不掺进去这条链路根本推不动。
  • 声明式触发器系统。 type / event / data / client / platform 五段式触发器协议、触发器的运行时执行器、 跳小程序 / 跳 URL / 曝光埋点 / 改其他组件 / 发自定义事件等一整套内置能力。
  • 推广与落地。 直接和运营、市场、产品团队对接 —— 收需求、演示、培训、兜底答疑。 这部分「非技术」的工作，在低代码项目里其实最决定成败。

简单讲：架构我拍板，核心链路前端 + 后端两边我都自己写。 带 3–5 人的小团队，随项目阶段弹性调整。 之所以同时管编辑器、运行时、发布链路和推广四件事—— 因为这类平台的关键决策只看一面都会走偏，得几面一起看才站得住。
