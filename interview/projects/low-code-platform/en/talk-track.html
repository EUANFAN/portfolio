Project Introduce
Interview version

This project was a low-code platform built for the operations team to create, manage, and launch campaign pages more efficiently.

Before we built it, most campaign pages had to be developed separately by engineers, which made the process slow, repetitive, and hard to scale.

With this platform, the operations team could use a visual editor to drag and drop components, configure page content, preview the result, and publish campaign pages without engineers.

Overall, it changed page delivery from repeated engineering work to a more configuration-based process, so the operations team could launch campaign pages much faster and support more business needs.

Project Background
Interview version

The project came from a real business need. The operations team had to launch campaign pages very frequently, and many of those campaigns were actually quite similar, but the old process was too slow and relied too much on engineers.

Before we built the platform, the workflow was mostly manual. The operations team would come up with the campaign idea, product and design would define the requirements, and then engineers had to build each page separately.

The main problem was that the whole process was slow, repetitive, and hard to scale. Since a lot of campaigns followed similar patterns, it did not make sense to rebuild them again and again. That was the main reason we decided to build a low-code platform.

Main Challenges
Interview version

On the business side, one big challenge was that different teams had quite different page requirements. If we built everything case by case, the platform would become very hard to maintain and scale.

On the product side, the key challenge was balancing flexibility and usability. We needed to support different business needs, but at the same time, the platform had to be simple enough for non-technical users like the operations team to use.

On the technical side, there were two main challenges. One was the visual editor, because different parts of it needed to stay in sync in real time. The other was page performance, because the platform was generating campaign pages for end users, so performance was very important.

Solution Design
Interview version

First, I introduced a unified data model, so page structure and content could be defined in a consistent way. I also set up a standard process for external component development and integration, so different business teams could extend the platform without breaking the overall structure.

Second, I built a visual editor to make page creation more intuitive and easier to use. I also prepared a user guide for the platform, which helped the operations team get started faster and reduced training and communication costs.

On the technical side, for the visual editor, I made the state flow more structured, so the left panel, the canvas, and the right config panel could stay in sync in real time. For the final pages, I focused on rendering and loading performance, because they were user-facing pages and performance was really important.

Technical Decision
Interview version

To support different business scenarios, I first introduced a unified page data model to describe page structure, content, and component configuration. In practice, it was basically a structured JSON object. The idea was that even if different teams had very different page requirements, we could still turn them into the same data structure first, and then pass that data to the rendering layer in a consistent way.

At the same time, to support more customized needs, I also defined a standard way for external components to be developed and integrated. Inside the page data model, all components were managed in an array-based structure and treated as the building materials of the final page. This allowed different business teams to extend their own component capabilities under the same platform rules, while keeping the core rendering logic consistent and easier to maintain.

For the visual editor, I used MobX to manage state across the left panel, the center canvas, and the right configuration panel. These three parts had to stay in sync in real time, so MobX worked well for this kind of reactive interaction. For example, when a component was selected from the left panel, the state would update, the canvas would re-render, and the right panel would show the related configuration options.

For performance, I used server-side rendering. Since the platform generated campaign pages for end users, first-screen loading speed and overall performance were very important. After a page was published, I used React's renderToString to generate the HTML, uploaded it to the CDN, and produced the final page link. This helped the page load faster and improved the user experience.
