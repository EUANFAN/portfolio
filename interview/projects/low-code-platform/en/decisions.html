06Key Decisions

The calls that shaped the platform

In interviews, this is the section I use to show judgment rather than just implementation detail. The pattern is simple: what problem we faced, what we chose, why we chose it, and what we knowingly gave up.

Decision 1 · MobX over Redux
Problem
The editor had deeply nested, high-frequency state changes across three panes.
Choice
One MobX StageStore with observable project, pages, widgets, layers, and selection state.
Why
The object graph was deep and highly interactive. MobX let us treat state like state, not turn every field edit into a mini state-management ceremony.
Cost
MobX felt more magical, and debugging was less explicit than Redux DevTools. We paid that cost and offset it with conventions and a separate history stack.
Decision 2 · Class Components over Hooks
Problem
The project started when Hooks were available, but the codebase leaned heavily on inheritance and MobX decorators.
Choice
Core editor components and widget wrappers stayed class-based.
Why
The widget runtime had a real base-class hierarchy, and the lifecycle timing was easier to reason about with explicit class methods.
Cost
It drifted away from mainstream React style and made future migration harder, but it sped up delivery at the time.
Decision 3 · Custom SSR over Next.js
Problem
We needed SSR, but the system was not “one app, one site.” It was a publish service rendering thousands of schema-driven pages with dynamic widgets.
Choice
We built the SSR path ourselves on top of Koa, jsdom, and renderToString.
Why
We needed full control over dynamic widget injection, browser simulation, and per-page schema-driven rendering. That fit our model better than bending a full framework around it.
Cost
We owned every edge case ourselves: cache isolation, SSR compatibility, fallback behavior, and external bundle execution safety.
Decision 4 · Three widget tiers instead of one loading strategy
Problem
Packing every widget into the client bundle would blow up the payload, but making everything runtime-loaded would hurt first paint.
Choice
Built-in widgets in the main bundle, internal widgets on CDN, external widgets on CDN with a stricter protocol.
Why
This split followed stability. Core primitives were always there; less common business widgets stayed decoupled and versioned.
Cost
The loading story became more complex, and moving a widget between tiers required operational discipline.
Decision 5 · No code editor for ops
Problem
Operators occasionally wanted non-standard interactions, and the fastest way to satisfy that would have been a custom JS panel.
Choice
We did not allow custom scripting. Interactions had to be expressed through declarative triggers.
Why
Giving non-engineers arbitrary code would have made security, auditing, debugging, and runtime evolution much harder.
Cost
Some one-off requests could not be solved instantly. We accepted that and turned repeated needs into proper trigger types.
Decision 6 · Fixed 375px editor canvas plus px2rem in production
Problem
The editor needed a deterministic WYSIWYG baseline, while the runtime still had to adapt to real mobile screens.
Choice
The editor used a fixed 375px canvas. Production translated the same styling into rem-based responsive rendering.
Why
Operators do not think in responsive design systems. A fixed reference width made the editing experience concrete and much easier to trust.
Cost
Extreme device widths were never perfect, and widget authors had to follow strict styling rules.
