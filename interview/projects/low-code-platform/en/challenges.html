07Challenges & Solutions

Five problems that were actually hard, and how I got them shipped

I've written this the way I'd walk through it in an interview. For each one I'll cover why it was hard, where the real pain was, how I broke it down, what I ended up building, the data structures behind it, and how I knew it worked. If the interviewer wants to dig deeper, I can keep going into the schema, the render path, or the publish pipeline.

01Unified schema: one engineering model for many campaign types

The hard part wasn't "store a JSON." It was designing one JSON shape that could describe very different campaigns and serve the editor, the preview, the publish flow, and the component system all at once.

Why this was hard

The first real problem in a low-code platform isn't drag-and-drop, it's how do you describe a page at all. If the schema is too thin, it can only describe static UI. If it's too heavy, the editor becomes a nightmare to maintain and the runtime gets complicated too. And we had to support a lot of different campaign shapes:

  • Some pages were one long scrolling page, others were multi-page flows.
  • Some widgets were plain atomic components, others were containers with their own inner panels.
  • Some content was static config, other content came from APIs at runtime.
  • Some interactions were page-level, others were widget-level behaviors, animations, or conditional visibility.
  • The same data had to be editable in the editor, renderable on the client, executable in SSR, and open enough for external widgets to plug in.

So my take was: this schema can't just describe "what the page looks like." It has to describe a full campaign project, including structure, layout, styles, behavior, animation, variables, data sources, and platform config.

The trade-offs I made
  • Layered modeling: Project / Page / Widget / Layer. Containers, pages, and widgets are different concepts, but the editor can operate on all of them the same way.
  • Business fields all live under widget.data. The platform only cares about the protocol, not the widget's internal business meaning.
  • Behavior, animation, and conditions are not mixed into the style block. That way the editor panel, the runtime executor, and SSR can each consume them independently.
  • Variables and data sources live inside the schema, not only at runtime. Otherwise preview, SSR, and client render can't stay aligned.
Why this model can carry very different campaigns
Unified Project / Page / Widget / Layer → Widget differences collapse into type + version + data → Interaction differences collapse into triggers / animations / condition → Data differences collapse into variableStore / dataBox → Editor, runtime, and SSR share one schema
How I'd say it in an interview

"The biggest step in this platform wasn't building the editor, it was getting the schema right first. I didn't think of it as 'a page JSON.' I treated it as a full campaign project: a Project contains Pages, a Page contains Widgets, and a container-type Widget can keep going with Layers inside it. On top of that tree I also pulled in behavior, animation, variables, data sources, and platform config."

"Once that was in place, the editor, the preview, the publish flow, and the external widget system could all work against the same contract. New business use cases didn't need changes to the platform core, just a new widget.type and a new shape of widget.data."

Why not just go with a simpler page JSON instead of this full Project / Page / Widget / Layer model?
Because we weren't building a static page renderer. We were building a full campaign project. On top of the UI tree it had to carry page switching, drilling into containers, widget behaviors, runtime variables, data sources, and publish metadata. A lightweight page JSON would have left gaps for the editor, preview, SSR, and external widgets very quickly. The engineering model is heavier, but it lets the whole pipeline move around one shared contract.
Why put business fields under widget.data instead of on the widget root?
Platform fields and business fields have to stay separate. Things like top, left, width, layout are platform concepts the whole system needs to understand. Things like url, courseId, teacherList only belong to a specific widget. Pushing them into data keeps the platform clean of business coupling, and lets external widgets reuse the same protocol.
If you redid this schema today, would you keep the same layering?
I'd keep the overall direction, but I'd document and version the schema much earlier. The shape itself is right. What got messy in the early days is that new fields kept showing up from business demands, and some of those field semantics were a bit improvised. If I did it again, I'd draw a sharper line between platform fields, business fields, and runtime fields, so that new teams integrating with the schema would pick it up faster.

02Editor core: three-pane collaboration and multi-level viewports

The real challenge wasn't drawing three panes on screen. It was getting the left rail, the center canvas, and the right inspector to all stay in sync with one editing context in real time.

Why this was hard

From the user's perspective it's a standard three-pane editor. Underneath there were several hard problems happening at the same time:

  • The left pane wasn't just a widget library. It also held pages and a layer tree, so the entry point itself had multiple modes.
  • The center canvas wasn't a single div. It was a multi-level viewport that moved between page, container, and layer contexts.
  • The right inspector had to change based on "current stage + number of selected items" at the same time.
  • Some widgets used flow layout, some used normal (freeform) layout, and the interaction models were totally different.

Without a shared editing-context model, the second you touched the left pane, the center and right could drift out of sync. Walking into a container and coming back out would leave state in a bad spot.

There's one more key piece I want to call out here: MobX actually became the real-time sync layer between the three panes. A lot of editor interactions aren't one-shot submits. They're continuously changing object state: the current page, the current selected widget, coordinates, prop values, the current viewport level. If I'd tried to keep those in sync with event buses or hand-written UI sync, the wiring would've been scattered and the state would've drifted fast.

How the editor looks through a MobX lens
// Shared editor state
class StageStore {
  @observable project
  @observable stages = []
  @observable swiperIndex = 0
  @observable activeTab = 'attribute'
}

// Left pane: switch page / layer
stageStore.setCurrentStage(page, 'page')

// Center pane: select widget / enter container / drag & resize
widget.isSelected = true
widget.modify({ left: 20, top: 40 })

// Right pane: edit properties
widget.modifyData('title', 'Fall Course')

// All three panes just @observer this tree. No extra UI sync events needed.
Why the center pane is a multi-level viewport

We didn't flatten every level into one canvas. Pages and layers are built as actual switchable viewports. So when you enter a container widget, you're not digging for a child node inside a huge DOM tree. You're walking into a new editing space.

Select container widget → StageStore.setCurrentStage(widget, 'widget') → Pick the default layer → StageStore.setCurrentStage(layer, 'layer') → WorkSpace slides into a new Swipeable viewport
Why I abstracted a Stage on top

Page, Layer, and container Widget are different types, but from the editor's point of view they behave very similarly. They all need to support adding children, removing, sorting, selecting, and deselecting. So instead of scattering if/else across business classes, I built a Stage abstraction that maps the children of the current context into one consistent list. 

class Stage {
  constructor(component, type) {
    this.component = component
    this.type = type
    this.list = component.pages || component.widgets || component.layers || []
  }

  addChild()
  removeChildren()
  selectChildren()
  sortChildren()
}

StageStore

Why MobX fit this editor especially well
  • The state is naturally a tree. Project, Page, Widget, Layer, plus the editor's own context stack, are all nested. MobX matches that shape directly.
  • Interactions are high-frequency and fine-grained. Selecting, dragging, resizing, switching pages, switching layers, editing props — it's constant. "Change the object, UI updates" is a better fit than dispatched actions here.
  • Three-pane collaboration is an observation problem. Left, center, and right don't need to know how each other is implemented. They all just observe the same state.
  • It keeps boilerplate low. With a heavier dispatch model, every interaction goes through action / reducer / selector. For an editor with this many small edits, the mental overhead would add up fast.
The interaction problems I specifically wanted to solve
  • flow and normal coexisting. flow is great for ops-driven sequential layouts, normal is great for freeform layout, and they have to share one canvas.
  • Entering and leaving a container. Walking into a container isn't a modal. It's a viewport switch with a breadcrumb, so the user always knows where they are.
  • Marquee and multi-select. Clicking empty space deselects. Box select only picks up widgets in the current stage, so it never accidentally grabs widgets from another layer.
  • Right-click context menu. Copy, cut, delete, bring-to-front, send-to-back, enter container — all go through the current stage, so the logic isn't scattered across the codebase.
How I'd say it in an interview

"On the surface this editor is a standard three-pane layout. The real challenge is that behind those three panes there has to be one shared editing context. I built a StageStore that holds the project, the current edit path as a stack of stages, a swiperIndex, and the active tab on the right. Switching pages on the left, dragging or resizing in the center, drilling into a container, editing properties on the right — all of that is really just reading and writing one stageStore."

"MobX was really important here. Three-pane sync isn't a 'who notifies who' problem. It's 'all three panes observe the same tree.' Page changes, selection changes, property edits, container drill-ins — those are all just mutations on observable objects. Left, center, and right are all observers and they update themselves, so I never needed to build a separate event bus to keep the UI in sync."

"On top of that, I abstracted Page, Layer, and container Widget children into a single Stage, so add / remove / update / select / sort all share one code path instead of if/else everywhere. And for entering a container, I used SwipeableViews to make it a real viewport switch instead of a modal, which is much easier for users to follow."

Why support both flow and normal layout? Why not pick one?
Because one isn't enough. Campaign pages have both sequential ops-style layouts and visually rich absolute-position layouts. Pure flow can't handle complex visual arrangements. Pure absolute positioning is a nightmare to maintain on long pages. The platform has to support both. Internally I just kept the two interaction models separated.
Why make entering a container a new viewport instead of expanding it inline?
Because once the container gets complex, expanding it inline makes selection, the layer tree, and drag boundaries really confusing. With a new viewport, the current stage has a clean boundary — the user always knows which layer they're editing, and the path back is obvious.
Why MobX + StageStore here instead of Redux or an event bus?
Because the core problem is high-frequency, fine-grained, cross-pane collaboration. An event bus can connect actions, but state ends up scattered. Redux can hold state, but the action / reducer overhead in a deeply nested tree like this is too much. What we actually needed was "change one object, all three panes react automatically." That's MobX territory, with the stageStore as the single source of truth.
What's the core value MobX is really giving you in this editor?
It's not just "a way to manage state." It fits an object-tree-driven editor particularly well. A lot of state already lives on objects — page.isSelected, widget.left, widget.data.title. With MobX, those local property changes are the UI update source directly. I don't have to invent a sync layer on top.

03Render convergence: keep the editor and the client as close to isomorphic as possible

The hard part of WYSIWYG is that the editor has an interaction shell and the client has a runtime shell, but the visuals and content still have to match.

Why this was hard

The editor and the client are fundamentally different environments:

  • The editor has selection frames, drag handles, resize handles, guide lines, context menus.
  • The client has touch events, scroll listeners, lazy loading, SSR hydration.
  • The editor canvas is a fixed size. The client has to adapt to any phone viewport.

If each side writes its own render logic, visual drift is just a matter of time. The answer can't be "add more tests." It has to be an architectural one: force both sides to share as much as possible.

Convergence architecture
Shared layer
WidgetClass getStyle / getAnimation / getWidgetContent
WidgetContent Always calls widgetConfig.onRender(ctx)
Container One flow / normal layout split, shared
Same .less files Styles live in one place
Editor side
BaseWidget + Rnd Drag, resize, selection state
Sortable flow list flow reordering
Client side
Widget wrapper Touch, lazy load, page behaviors
hydrate / render Takes over SSR output or does pure client render
The convergence strategy I used
  • One render entry. The actual widget content is always produced by widgetConfig.onRender(ctx).
  • One style computation. Base styles — background, border, radius, shadow — are computed in the shared base class.
  • One container implementation. Page and layer reuse the exact same Container render path.
  • Differences stay in the wrapper. The editor wraps with an interaction shell. The client wraps with a runtime shell.
  • Fixed editor canvas + client px2rem. The editor shows a 375 baseline. The client scales proportionally to the viewport.
The render path
WidgetSchema(type/version/data) → getWidgetConfigByType() → WidgetClass.getWidgetContent() → WidgetContent → widgetConfig.onRender(ctx)
Engineering details worth calling out in an interview
  • autoHeight widgets. Some widgets don't ship with a fixed height. The editor has to measure the real height and write it back into the model, otherwise selecting it later makes the height "jump."
  • flow / normal consistency. It's not just widget content — the split rules between the two layout modes inside a container have to match on both sides.
  • Runtime variables count as content too. Page titles, dynamic text, variables fed from data sources all have to work in SSR as well. They can't be a client-only story.
How I'd say it in an interview

"I don't think you can solve editor / client consistency with more testing. It has to be pushed into the code structure. What I did was force everything that should be identical into a shared layer: shared WidgetClass, shared WidgetContent, shared Container, shared onRender, shared styles. Each side only differs in its wrapper — the editor one handles drag and selection, the client one handles touch and hydrate."

Why not just maintain two render code paths and optimize each for its environment?
Short term it feels faster. Long term it almost always goes sideways. The whole point of a low-code platform is that it keeps onboarding new widgets over time. If both sides have their own renderer, every new widget adds two places to keep aligned. The debugging cost passes the cost of shared design pretty quickly.
How did you handle widgets that "look right in the editor but render at the wrong height in production"?
Usually it's autoHeight widgets, rich text, or image aspect ratios. The fix was to keep height calculation in the shared layer and the editor wrapper. When needed, the editor measures the real height and writes it back into the model, so the visual size isn't only living in the DOM while the schema thinks something else.

04Component platformization: a widget protocol with dynamic loading

The platform can't ship a release every time a new business widget appears, but it also can't let external widgets break older campaign pages. So widgets had to become a real runtime system, with versions.

Why this was hard

Component platformization usually runs into three conflicts at the same time:

  • Business teams want to ship their own widgets without waiting on a platform release.
  • The platform still wants every new widget to automatically show up in the editor, the properties panel, the client runtime, and SSR.
  • Campaign pages that have already shipped are live in production, so upgrading a widget must never break an older page.

Internal widgets
PageData.innerWidgets Platform-owned but loaded on demand
CDN index.js / index.css Fetched at runtime
External widgets

Unified entry

The widget contract
export default {
  name: 'CourseCard',
  type: 'CourseCard',
  version: '1.2.0',
  category: 'widget',
  data: {
    title: '',
    image: '',
    jumpUrl: ''
  },
  config: {
    title: { text: 'Title', type: 'Input' },
    image: { text: 'Cover', type: 'FilePicker' },
    jumpUrl: { text: 'Jump URL', type: 'Input' }
  },
  methods: {},
  listeners: {},
  onRender(ctx) {
    return <div>...</div>
  }
}
Why this contract matters
  • data defines the default business data for a widget instance.
  • config defines how the right-side properties panel is rendered. The platform doesn't have to hand-write a form for every widget.
  • onRender is the one shared content output, used by both sides.
  • methods / listeners give the widget reusable capabilities and runtime event handlers.
Dynamic loading flow
Editor or preview sees widget.type / version → loadScript(CDN index.js) → loadStyle(CDN index.css) → Widget self-registers into god.__components__ → It's merged into WidgetConfigs and used like any built-in
The two things I specifically wanted to get right
  • Version stability. Historical campaign pages already have a version in their schema, so the runtime always fetches that exact version. A new widget release can't leak back and change an older page.
  • Loading reliability. Script and style loading have caching, timeouts, retries, and error logging, so ops doesn't hit "widget randomly doesn't load" in production.
How I'd say it in an interview

"I made widgets a protocol-driven runtime registry. Widget authors don't touch the platform repo. They publish a bundle to the CDN following the data / config / onRender contract, and the platform picks it up as a first-class widget in both the editor and the runtime. And because the version is pinned inside the schema, historical campaign pages always pull the exact version they were published with. New releases can't accidentally overwrite older pages."

Why runtime dynamic loading instead of bundling every widget into the main bundle?
Because in a platform, the widget count only grows. Bundling everything would keep making the editor and preview heavier, and every new widget would need a full platform release. The platform would become a bottleneck for every business team. Dynamic loading really just decouples widget releases from the platform release.
Why CDN + version instead of managing external widgets through npm?
npm is better for build-time dependencies. Here we have to decide at runtime which widget version to load, per campaign page. CDN + version fits the "one platform, many live historical pages" model much better, because each page can pin the exact version it was published with, and it doesn't need to move when the platform moves.

05Publish pipeline: SSR pre-render with graceful fallback

The hard part wasn't renderToString. It was doing it on the server for schema-driven pages with external widgets, runtime variables, and template assembly — and keeping the publish itself rock solid.

Why this was hard

Campaign pages care a lot about first-paint speed. Pure client-side render hurts here. But once you go down the SSR path, you hit a few problems fast:

  • Campaign pages aren't a fixed site. They're a large number of schema-driven dynamic pages.
  • Widgets aren't all in the main repo. The server only knows which external widgets and versions to load after reading the schema.
  • Some widgets are SSR-safe. Others will crash the moment they touch a DOM API.
  • The publish flow can't fail just because one widget's SSR pass threw.

Graceful degradation
  • Widget not SSR-safe: skip it on the server instead of blowing up the page.
  • Preload step fails: set needHydrate to false directly.
  • renderToString throws: still generate the page, just fall back to client render.
  • Project in an odd state (not started or ended): also disable hydrate to avoid inconsistent takeover.
The designs I'm most proud of in this pipeline

Preload by widget version

Clearing require.cache

HTML + JSON dual output

How I'd say it in an interview

"On the publish side, I treated SSR as a first-paint booster, not the only path. The server scans the schema for the widgets and behaviors it actually uses, pulls the SSR-capable external widget bundles, registers them inside a jsdom environment, then runs renderToString to produce the main HTML. If that works, the client hydrates. If it doesn't, it falls back to client render. Either way, the publish itself stays stable."

"One thing I really cared about was module isolation. Node's require cache means it's easy to leak state between publishes if you don't clean it up. So before each SSR pass, I clear the relevant module cache, so a previous publish's widget code doesn't bleed into the next one."

Why not use something like Next.js for SSR?
Because we weren't building a fixed site. We were building a publish service for a large number of schema-driven campaign pages with runtime-loaded widgets. We needed full control over widget scanning, external bundle injection, template assembly, CDN uploads, and failure fallback. Building on top of Koa + jsdom gave us more room than an all-in-one framework would have.
If SSR fails, why not block the publish?
Because publish success matters more than SSR. SSR is there to improve first paint, not to become a single point of failure. If one widget's SSR compatibility could block an entire campaign launch, the business would never accept it. My rule was: SSR is an enhancement, client render is the real safety net.
Why run external widget code on the server at all? What's the biggest risk?
Because without running the widget's registration, SSR has no onRender for it, and the page can't fully pre-render. The main risk is that widget code isn't always pure — some of it touches browser APIs or has heavy side effects. So we only SSR widgets that meet the constraints, and we keep a full fallback path so that this step can never become a single point of failure for publishing.
