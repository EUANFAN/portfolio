Low-Code Campaign Platform / Core Modules
05Core Modules

The five parts that carried most of the engineering weight

The platform looked like one product, but the engineering depth was not evenly distributed. These were the five areas where most of the real architecture work happened.

01 · Editor core One shared MobX state tree

The visible layout was a classic three-pane editor. The interesting part was how the three panes stayed in sync. We used one MobX-driven StageStore instead of event buses or Redux-heavy plumbing.

Model
StageStore → Project → Pages → Widgets → Layers, all restored as observable objects.
Communication
The left panel, canvas, and settings panel were all @observer views over the same state tree.
Pain point
MobX 4 does not track dynamically added object fields the way people expect, so we wrapped widget data updates to force safe reference replacement.
Why it mattered
This kept editor interactions direct and low-friction even with nested objects and many small updates.
02 · Canvas architecture Flow layout and freeform placement together

Campaign builders needed two mental models at once: “stack content from top to bottom” and “place this exactly here.” We supported both on the same canvas.

Flow layout
Used for list-like content, built with sortable interactions and document-flow ordering.
Freeform layout
Used for floating buttons, decorations, and precision layout, built on drag + resize.
Multi-level viewport
Users could drill from project to page to container layer as separate editing spaces instead of one huge DOM tree.
Workspace tooling
Rulers, selection box, and viewport switching turned the canvas into an actual editing surface rather than a plain preview area.
03 · Widget platform Built-in, internal, and external tiers

Widgets were the platform’s most active asset, so we needed a model that balanced stability with extensibility.

Built-in widgets
Core primitives like image, text, and container shipped in the main bundle.
Internal widgets
Business-specific widgets were split into their own bundles and loaded from CDN at runtime.
External widgets
Other teams could build and publish widgets through a scaffolded CLI flow and versioned CDN contract.
Unified protocol
Every widget exposed the same shape: type, version, data, config, methods, and onRender.
04 · Declarative triggers No custom scripting for ops

We were intentional about not giving operators a “write your own JS” escape hatch. Instead, interactions were expressed as event + action declarations.

Events
onClick, onShow, onScroll, and custom events covered most campaign interactions.
Actions
Redirects, scroll, cross-widget updates, share behavior, and pub/sub style custom events.
Tradeoff
This was less flexible than arbitrary scripting, but far safer for security, debugging, and long-term maintenance.
05 · SSR publish path Fast paint with graceful fallback

This was the densest part of the system. It involved browser simulation on the server, external widget loading, React SSR, asset assembly, and failure fallback.

Server simulation
jsdom provided a browser-like environment so widget code could execute during SSR.
Widget preload
Used schema metadata to fetch widget bundles from CDN and register them before rendering.
HTML output
Rendered React to string, injected HTML and PageData into a template, then pushed the result to CDN.
Fallback
If SSR failed at any point, the page still published and rendered client-side. Shipping the page mattered more than preserving SSR.
