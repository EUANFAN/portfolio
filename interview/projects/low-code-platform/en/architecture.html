04Architecture

The layers, the stack, and the flows that mattered

The easiest way to understand the platform is as four layers: the editor on top, a shared schema in the middle, a Node/Koa publish service behind it, and a consumer runtime that renders the final campaign page.

Flow 1: adding a widget to the canvas

Pick a widget from the left panel → Stage.addChild(widget) → Push into an @observable list → @observer canvas re-renders → New widget appears immediately

The same path is used when the right panel edits widget data. That was one of the big benefits of the MobX model: the three surfaces were not “syncing UI” with each other. They were simply reacting to the same state tree.

Flow 2: publish to production

Ops clicks publish → Schema hits the Koa API → Node boots jsdom → Load external widget bundles → renderToString(<App />) → Assemble HTML + PageData → Push HTML and JSON to CDN

Flow 3: widget upgrade without breaking old pages

Ship widget v2 → Upload to /components/{type}/{version} → Old schema still pins v1 → Historic pages stay untouched

The version lived in the schema, not in some global “latest” registry. That was a simple choice, but it made the platform much safer.
