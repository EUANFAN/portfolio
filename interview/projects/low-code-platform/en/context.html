Low-Code Campaign Platform / Context & Problem
02Context & Problem

Why we built it in the first place

What the world looked like before the platform

  • Every campaign page was still hand-built by frontend engineers. Even a fairly simple page usually took 1-2 days from request to launch.
  • Common building blocks like hero images, rich text, fixed CTA buttons, and signup modals were rewritten again and again, so code quality and styling drifted over time.
  • Multi-channel delivery was painful. The same campaign often had to work in mobile web, WeChat H5, app WebView, and Mini Program WebView, each with slightly different runtime behavior.
  • Tracking, redirects, share config, and mini-program jumps were implemented differently from project to project, which created both data inconsistency and debugging overhead.
  • The frontend team was stuck in a delivery loop. Anything more strategic, like a widget ecosystem, SSR, or better tooling, kept getting pushed out by campaign work.

What that caused in practice

Ops moved too slowly
Simple changes like swapping copy or changing a banner still needed a frontend slot, so fast-turn campaigns were hard to execute.
Frontend time was wasted
A lot of engineering effort was going into repetitive layout work instead of reusable platform capability.
Quality was inconsistent
Visual polish, event tracking, and cross-channel behavior depended too much on people remembering conventions.
Risk kept accumulating
There was no version discipline for campaign components, so a “small” shared change could accidentally affect older live pages.

What problem we were actually solving

On the surface, this looked like “build an internal editor.” But the real problem was delivery mode. We wanted to move campaign production from “every page needs frontend engineering” to “ops can assemble and publish pages themselves, while the platform guarantees quality”.

Buying an off-the-shelf SaaS tool was not a great fit either. The education business had heavy customization around data, enrollment flows, teacher cards, referral widgets, and share behavior. In practice, a generic page builder would still need a lot of custom engineering, so we chose to build it ourselves.

What the frontend architecture had to guarantee

  • It had to scale with business variation. The schema and widget system needed to support new campaign types without rewriting the platform core.
  • What you see in the editor had to match production closely. That meant a fixed editor canvas and a runtime that could translate the same styling to real devices.
  • Performance was not optional. These pages were traffic entry points, so SSR and CDN delivery were baseline requirements, not nice-to-haves.
  • Publishing needed a fallback story. If SSR failed, the page still had to ship and still had to render correctly on the client.
  • Ops could not be writing JavaScript. Interactions had to be expressed through declarative triggers, not custom scripts.
  • A small team had to be able to keep it alive. The design had to stay understandable with 3-5 engineers, not depend on tribal knowledge.
